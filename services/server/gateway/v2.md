# Gateway v2 Design

Gateway v2는 public edge 동작만이 아니라 품질 게이트를 계약 수준으로 끌어올리는 것을 포함한다.

## 목표
- `gateway-service` 구현 repo에 JaCoCo coverage report와 coverage verification을 추가한다.
- `gateway-service` 구현 repo에 k6 기반 부하 테스트 시나리오를 추가한다.
- 품질 게이트는 "있으면 좋은 도구"가 아니라 v2 기준 CI/CD 계약의 일부로 취급한다.

## JaCoCo 품질 게이트

v2에서 Gateway CI는 단순 `test` 성공만으로 끝내지 않는다.

Required behavior:
- `jacocoTestReport`를 생성한다.
- `jacocoTestCoverageVerification`으로 최소 coverage threshold를 검증한다.
- threshold 미달 시 `test` stage 또는 동등한 품질 게이트 단계에서 실패한다.
- coverage 기준은 `app`, `common` 모듈의 실제 런타임 코드에 적용한다.
- generated artifact, contract fixture, boilerplate launcher는 제외할 수 있다. 제외 규칙은 구현 repo에 명시한다.

Contract baseline:
- line coverage gate를 기본으로 둔다.
- branch coverage gate는 선택이 아니라 v2 기본선에 포함한다.
- threshold 값은 repo가 임의로 낮추지 않고, 변경 시 service-contract 문서와 구현 repo를 함께 갱신한다.

## k6 부하 테스트 게이트

v2에서 Gateway는 edge 서비스 특성상 최소한의 HTTP 부하 회귀 검사를 가진다.

Required behavior:
- k6 스크립트를 구현 repo에 보관한다.
- public route 기준 smoke/load 시나리오를 분리한다.
- smoke 시나리오는 CI 또는 pre-release 검증에 연결할 수 있어야 한다.
- sustained/spike 시나리오는 nightly 또는 release candidate 검증에 연결할 수 있어야 한다.

Minimum scenario set:
- `GET /health`
- `POST /v1/auth/login`
- 인증이 필요한 representative route 예: `GET /v1/users/me` 또는 `GET /v1/documents/{id}`
- 관리자 검증 representative route 예: `/v1/admin/**`

Required assertions:
- error rate threshold
- p95 latency threshold
- upstream timeout/5xx 급증 감지
- login rate-limit 경계가 의도대로 유지되는지 확인

## CI/CD 반영 기준

v2 기준 CI contract는 아래 흐름을 권장한다.

1. `contract-lock`
2. `setup-runtime`
3. `test`
4. JaCoCo coverage verification
5. build
6. image
7. pre-release k6 smoke

Notes:
- JaCoCo는 PR 기준 필수 게이트로 둔다.
- k6 smoke는 적어도 release branch, protected branch merge 전, 또는 수동 release gate에서 실행 가능해야 한다.
- 장시간 soak/spike profile은 일반 PR CI에 강제하지 않는다.

## 운영상 이유
- Gateway는 여러 downstream 실패가 edge latency와 error rate로 바로 드러나는 서비스다.
- 따라서 v2 기준 품질 게이트는 기능 회귀뿐 아니라 coverage와 load 회귀를 함께 본다.
- JaCoCo는 코드 경로 회귀를, k6는 edge behavior 회귀를 잡는 최소선이다.

## Contract Notes
- 이 문서는 v2 목표 계약이다. 현재 구현이 아직 JaCoCo gate 또는 k6를 포함하지 않아도 된다.
- 구현 repo가 v2 범위로 전환되면 CI workflow, Gradle 설정, 테스트 자산, release runbook을 함께 갱신한다.
